Hypothetical Stress Scenario Engine

Mo Minoneshan
2026
GitHub Demo

Overview

The Hypothetical Stress Scenario Engine is a production-grade cross-asset scenario generator built for RBC Capital Markets summer 2026. Given a set of user-specified "pinned" factor shocks (e.g., "S&P 500 +20%, 10Y rates +87bp"), the engine computes coherent, statistically-grounded implied shocks across all other risk factors using conditional Gaussian methodology.

Core Innovation: Rather than ad-hoc scaling or rule-of-thumb spillovers, the engine respects the covariance structure of Bloomberg futures data across 30+ years, then applies multicollinearity diagnostics to warn when scenarios violate historical market regimes.

Key Features

  • Bloomberg Native: All 44 tickers sourced from Bloomberg (equity, rates, commodities, FX futures)
  • Tier A Screening: Only factors with 90%+ data completeness + 10+ years history qualify for MVP
  • Three Covariance Methods: Sample, EWMA (recent regime emphasis), Ledoit-Wolf (shrinkage for stability)
  • Unit-Aware: Automatic handling of rates (price changes in bp) vs commodities/equity (returns)
  • Multicollinearity Audit: Pinned shocks checked against condition number, high correlations, magnitude extremes
  • Interactive Dashboard: Preset scenario templates + custom factor pins via ipywidgets

Motivation

Traditional stress scenarios at financial institutions often rely on:

  • Ad-hoc Scaling: "If equities drop 20%, then bonds move +500bp" (rule-of-thumb, no covariance)
  • Isometric Shocks: "All risk factors move the same percentage" (ignores correlations)
  • Expert Override: "We know what should happen" (no statistical grounding, hard to audit)

These approaches miss spillover effects and produce scenarios that violate historical covariance relationships.

Conditional Gaussian Solution

By conditioning on a subset of "pinned" factors, we can estimate the joint distribution of all remaining factors using the multivariate Gaussian framework:

μB|A = μB + ΣBA ΣAA-1 (shockA - μA)

This respects covariance, is repeatable, auditable, and supports validation against historical stress episodes.

System Architecture

The engine is organized into seven layers:

Layer Responsibility Status
1. Data Layer Bloomberg data ingestion, cleaning (ffill, holiday handling) Complete
2. Screening Tier A universe selection (completion ≥90%, years ≥10) Complete
3. Transforms Per-asset-class: equity/commodity/FX → pct_change, rates → diff() Complete
4. Statistics Covariance estimation (sample, EWMA, Ledoit-Wolf) Complete
5. Conditional Gaussian Core engine: condition on pinned shocks, compute implied means Complete
6. Validation Multicollinearity checks, sanity tests, historical replay Complete
7. Application CLI, Streamlit dashboard, Jupyter notebook UI (ipywidgets) Phase 2

Risk Factor Universe

The engine ingests 44 Bloomberg tickers covering six asset classes. All data sourced from Bloomberg Terminal, June 2026 pull.

Asset Classes

  • Equities (4 tickers): ES1 (S&P 500), NQ1 (Nasdaq-100), VG1 (Russell 2000), NO1 (Nikkei 225)
  • Rates (6 tickers): TY1 (US 10Y), FV1 (US 5Y), TU1 (US 2Y), RX1 (Bund 10Y), DU1 (Schatz 2Y), ER1 (Euribor 3M)
  • Energy (9 tickers): CL1 (WTI crude), NG1 (nat gas), CO1 (CO2), TZT1 (TTF gas), FN1, QS1, XB1, HO1, ZEA1
  • Metals (10 tickers): GC1 (gold), SI1 (silver), HG1 (copper), PL1, PA1 (palladium), LA1 (aluminum), LX1, LN1, LL1, LP1
  • Agriculture (2 tickers): W 1 (wheat), C 1 (corn)
  • FX (3 tickers): EC1 (EUR/USD), JY1 (JPY/USD), CD1 (CAD/USD)

Tier A Screening & Universe Definition

Not all 44 tickers are suitable for MVP. The engine uses a four-tier classification:

Tier Criteria Count Usage
A Completion ≥90% AND Years ≥10 28 tickers MVP fit
B Completion 50–90% OR Years 5–10 8 tickers Phase 2 extension
C Years <5 or very low completion 5 tickers Exclude (insufficient history)
D No data / completely missing 3 tickers Skip

Tier A Universe (28 Factors)

The MVP engine fits on this subset. These represent liquid, long-history benchmark futures across all asset classes.

Data Transforms & Missing-Value Policy

Per-Asset-Class Returns

Different asset classes require different transformations to achieve stationarity. Confirmed with RBC risk team (June 18, 2026):

Asset Class Transform Rationale
Equities, FX, Commodities pct_change() Price-based returns; stationary
Rates & Bonds diff() Level changes in contract price (proxy for bp moves)

Holiday Missing-Value Policy

Bloomberg data spans multiple exchanges (CME, EUREX, LME, ICE). Holiday calendars differ → NaN patterns.

  • Strategy B (Recommended): ffill(limit=3) then dropna() → 5,178 rows (82% retention)
  • Forward-fill up to 3 days (handles cross-exchange holiday misalignment), then drop rows with residual NaN.

Covariance Estimation

The covariance matrix Σ is central to the conditional Gaussian engine. Three methods available:

1. Sample Covariance

Standard unbiased estimator. Pros: No hyperparameter tuning. Cons: Can be ill-conditioned for 28 factors.

2. EWMA Covariance

Exponentially-weighted moving average with decay λ = 0.94 (RiskMetrics standard). Captures recent regime shifts effectively.

3. Ledoit-Wolf Shrinkage

Shrinkage estimator that trades off bias for lower variance. Guarantees positive-definiteness, lower condition number.

Fit Window

MVP uses 2010-01-01 → present (4,821 rows). Covers post-GFC era, low-rate period, COVID shock, and 2022 rate spike.

Conditional Gaussian Engine

Engine Workflow

  1. Input: User specifies pinned factors with shocks
  2. Validate: Check units (return vs bp), validate shock reasonableness
  3. Multicollinearity Check: Warn if condition number is high or pinned factors highly correlated
  4. Invert: Compute inverse covariance with ridge regularization if near-singular
  5. Compute: Apply conditional mean formula for free factors
  6. Uncertainty Bands: Compute p5–p95 from conditional variance via Gaussian quantiles
  7. Output: Table of implied shocks by factor, with warnings and diagnostics

Multicollinearity Checks

Three diagnostic layers run before outputting results:

Layer 1: Direction Conflicts

Check if pinned factors have correlation |r| > 0.75 AND shocks in opposite directions (unusual regime).

Layer 2: Magnitude Extremes

Check if any shock falls in historical tail (> 4σ triggers soft warning, > 6σ triggers hard warning).

Layer 3: System Fragility

If condition number > 1000, system is fragile—suggest re-checking pinned factors or using different covariance method.

Validation & Sanity Checks

Historical Replay Testing

Validate against known stress episodes (2022 rate shock, COVID crash):

  1. Fit covariance on data before event
  2. Pin historical shocks from event date
  3. Compare generated implications to actual realizations
  4. Compute error metrics (MAE, RMSE, direction accuracy)

Sanity Tests

  • Spillover Direction: Equities down → credit spreads up?
  • Magnitude Proportionality: Larger pins → larger implications?
  • Zero Shock Identity: Zero pin → zero implication for uncorrelated factors?

API & Interface

Python API

from rbc_scenario.engine import ConditionalGaussianEngine

engine = ConditionalGaussianEngine(
    prices_df=prices_clean,
    returns_df=returns_df,
    transform_map=transform_map,
    cov_method='ewma'
)

scenario = {
    'ES1 Index': {'shock': 0.20, 'unit': 'return'},
    'TY1 Comdty': {'shock': 0.0087, 'unit': 'price_diff'},
}

result = engine.run(scenario, validate=True)
print(result.summary_table())
